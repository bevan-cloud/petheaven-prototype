Ratings & Reviews Enhancement

A two-phase initiative to build purchase trust, increase conversion, and differentiate Pet Heaven in the SA ecommerce market through improved review visibility, verified purchase signals, and intelligent review surfacing.

Platform: Magento Delivery: 2 phases Moderation: Manual pre-publish Guest reviews: Not allowed Version 2.1 — All decisions locked except verified reviews path
Problem statement

Reviews on petheaven.co.za are functional but underperforming as trust and conversion tools. The reviews tab sits third in the page navigation hierarchy — below the fold and behind two clicks — meaning the majority of customers never encounter review content during their purchase decision.

There is no mechanism to surface review quality, no purchase verification, and no review collection pipeline. The review corpus grows slowly, lacks credibility signals, and is architecturally invisible at the moments it matter most.

Pet food and pet care purchases are high-consideration, repeat-purchase categories. Customers actively seek peer validation before committing. Competitors in the SA ecommerce space have not raised the bar here — Pet Heaven has an opportunity to lead.

Phase 1 — Acceptance criteria

Feature 1 — Star ratings on listing & search
  • Star rating (1–5) and review count display on product card for all products with at least 1 approved review
  • No rating element renders for products with zero approved reviews — no empty state shown
  • Renders correctly on both category listing and search results pages
  • Rounding logic is consistent with PDP star display
  • Renders correctly on mobile and desktop viewports
Feature 2 — Above-fold review snippet
  • Highest-rated snippet surfaces the approved review with the highest star rating; ties broken by most recent
  • Most recent snippet surfaces the most recently approved review
  • Both snippets can be enabled/disabled independently in Magento admin config
  • Minimum review count threshold is configurable in admin; section hidden below threshold
  • Snippet body truncates at 2 lines with inline "Read more" expand — no page navigation required
  • Snippets appear above the Description / Additional Info / Shipping tab bar
  • Star rating / review count anchor scrolls to the reviews section
  • Verified badge displays on snippet if review is verified
Feature 3 — Post-purchase review email
  • Email triggers automatically on order status "Complete" with 5–7 day configured delay
  • One email sent per order regardless of item count; lists all products with individual review CTAs
  • Email suppressed if customer has already reviewed all products in the order
  • Email suppressed for opted-out customers
  • RFM segment data passed correctly to email platform for personalisation
  • All three fields required at submission: star rating, review title, review body
  • Approval notification email sends when moderator approves a submitted review
  • Email renders correctly across Gmail, Outlook, Apple Mail
  • CTA links directly to PDP with review form in view; login prompt shown if session is inactive

Phase 2 — Acceptance criteria

Feature 4 — Verified reviews
  • Review submission form is identical for all customers — no variation based on order status or verification eligibility
  • No messaging, prompt, or UI element during submission references verification status
  • Reviews submitted against a completed order are badged "Verified purchase" post-moderation
  • Reviews submitted while order is in processing are published unverified; badge applied automatically when order completes
  • Full refund on the reviewed SKU removes the verified badge; review remains published
  • Partial refund on SKU does not affect verified status
  • Refund reversal reinstates the verified badge automatically
  • Retroactive pass re-checks all existing reviews against order history on launch
  • Multiple reviews per customer per product are each independently assessed for verified status
  • Unverified reviews are not hidden — badge absence is the only distinction
  • Verified status is stored, queryable, and used as a signal in relevance sort
Feature 5 — Mark as useful
  • Logged-in customers can mark any review as useful except their own
  • One vote per customer per review; vote can be toggled off
  • Useful count updates immediately on vote (optimistic UI)
  • Useful count stored and queryable as a relevance sort signal
  • Logged-out users see vote counts but cannot interact — clicking prompts login
  • Customers cannot vote on reviews they authored
Feature 6 — Relevance sort & customer filters
  • Default relevance sort applies on page load using four signals: star rating, usefulness count, verified status, recency (3-month decay)
  • All four signal weights configurable in Magento admin without code deploy
  • Recency applies a decay function boosting reviews created within the last 3 months
  • Customer filters — star rating, verified only, most recent, most useful — individually accessible and combinable
  • Filter state persists within session; cleared on new session
  • Sort degrades gracefully when verified or usefulness data is sparse — must not produce worse order than current default

Constraints & dependencies

What needs to be in place

TechAll features implemented within the Magento stack. Third-party integration (Option B) requires a separate technical evaluation before Phase 2 scoping.
TechVerification state machine requires three order status event hooks: completed, refunded, and refund reversed. Technical feasibility to be confirmed with dev team before Phase 2 planning.
TechPost-purchase email requires confirmed integration between Magento order events and the email platform. Exact send delay timing to be confirmed with ops team.
BusinessRFM segment-to-message mapping must be supplied by the CRM team before Phase 1 email development begins. This is a Phase 1 blocker.
BusinessVerified Reviews implementation path (native vs third-party) must be decided before Phase 2 development scoping. Delay to this decision delays Phase 2 start.
UXModeration queue capacity must support increased submission volume before the post-purchase email goes live. More reviews means more moderation load.
UXRelevance sort must degrade gracefully in low-volume scenarios — a product with few reviews must not sort worse than under the current default sort.
