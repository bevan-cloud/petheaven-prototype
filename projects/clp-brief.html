Category Landing Page
Enhancements

A two-phase initiative to transform Pet Heaven's category pages from bare product grids into high-converting, editorially curated shopping surfaces — with hero zones, subcategory navigation, brand spotlights, and SEO content that drives organic acquisition.

Platform: Magento Surface: Category pages Delivery: 2 phases Version 1.0
Problem statement

Category pages on petheaven.co.za function purely as filtered product grids. A customer landing on "Dog Food" is met immediately with a paginated list of products — no editorial context, no navigation to subcategories, no brand groupings, no seasonal promotions, no guidance to help them choose. The page does no selling work.

For new or unfamiliar visitors — who may know they want dog food but not which brand, size, or life stage — the current category page creates decision paralysis rather than resolving it. There is no hierarchy to guide them (dry vs wet, puppy vs adult, premium vs value), no featured products to anchor their attention, and no trust signals to reassure them they are in the right place.

From an SEO perspective, category pages are frequently the highest-traffic organic entry points in ecommerce. A bare grid with no unique written content, no structured heading hierarchy, and no internal link signals leaves significant ranking potential on the table — particularly for high-value queries like "dog food South Africa", "cat food subscription", or brand-level queries where Pet Heaven should rank.

Acceptance criteria

1.1 — Category hero banner

  • Banner renders at the top of the category page, above the filter bar and product grid
  • Image, headline, subtext, and optional CTA button are individually configurable via CMS (Magento page builder or block)
  • Banner is responsive: full-width on desktop, correct crop / reduced height on mobile
  • If no banner is configured for a category, the section does not render and the grid starts immediately
  • Banner image is served as WebP with appropriate alt text for accessibility and SEO

1.2 — Subcategory tile navigation

  • Tiles are auto-generated from the active child categories of the current category in Magento's category tree
  • Each tile links to the child category URL and displays the category name
  • On mobile, the tile strip is horizontally scrollable with no horizontal page overflow
  • If a category has no children, the tile strip does not render
  • Optional tile thumbnail image is supported but not required

1.3 — Featured product row

  • Featured products are selected via a Magento CMS widget or category attribute — not hardcoded
  • The row supports 4–6 products and uses the same product card component as the main grid (consistent styling)
  • A "Featured" or "Top picks" section heading is shown above the row
  • If no featured products are configured, the row does not render
  • On mobile, the row is horizontally scrollable

1.4 — SEO content block

  • Block renders at the bottom of the page, below the product grid and pagination
  • Content is editable per category via CMS (Magento page builder)
  • On mobile, content is collapsed to ~3 lines with a "Read more" toggle
  • Content is visible to search engine crawlers (not hidden via CSS display:none)
  • If no content is configured, the block does not render
Key decisions
Hero banner implementation — CMS Block vs Page Builder widget Open

Magento supports multiple mechanisms for category page content injection. The choice of CMS Block (simpler, dev-configured) vs Page Builder widget (visual editor, marketing-configurable) affects both implementation complexity and the content team's day-to-day workflow.

Option A — Page Builder category attribute (recommended)
Marketing team can manage banner independently
Native to Magento — no additional modules
Per-category control out of the box
Page Builder has a learning curve for non-technical editors
Magento 2.4+ only — verify version compatibility
Option B — Named CMS Block per category
Simpler to implement and maintain in code
Full HTML/CSS control in the block
Requires dev to create a new block per category
Content team cannot self-serve on updates
Subcategory tile data source — category tree vs manual Locked

Subcategory tiles will be auto-generated from the Magento category tree (child categories of the current category). Manual tile management would create a maintenance burden and fall out of sync with the actual category structure over time. Auto-generation is the correct pattern.

SEO block visibility — server-rendered vs JS-loaded Locked

The SEO content block will be server-rendered (visible in HTML source) and collapsed via CSS on mobile using a standard details/summary or max-height pattern. JavaScript-loaded content risks being missed by crawlers; server-rendered content with CSS collapse is the correct implementation for SEO intent.

Constraints & dependencies
Tech The Magento version must be confirmed before committing to Page Builder for the hero banner. Page Builder is available from Magento 2.3.1 Commerce and 2.4.3 Open Source. If on an older version, the CMS Block approach (Option B) must be used.
Business Category hero images do not currently exist. Marketing must produce category-level imagery before Phase 1 can go live. Each top-level category requires at minimum a desktop and mobile crop. Estimate: 6–10 categories × 2 crops = 12–20 image assets.
SEO SEO content copy must be written for each category before the block can be populated. This is a content team deliverable, not a development deliverable. A brief per category (target keyword, tone, ~150 words) should be prepared ahead of Phase 1 launch.
UX The filter panel and product grid layout must be audited before adding the hero, tile strip, and featured row above the grid. On mobile, each new section pushes the product grid further below the fold — a layout budget must be agreed to avoid the grid being unreachable on first scroll.
Tech Phase 2's smart sort defaults require access to sales velocity data (orders per product in a rolling window). This data must be queryable from Magento or exportable from the analytics/BI layer. Confirm data availability before scoping Phase 2.
Business Phase 2's brand spotlight tiles require a brand taxonomy to be established and maintained in Magento. If brands are not currently structured as filterable attributes or categories, a data migration and attribution task must be completed first.

Development effort estimate

Phase 1 is template-layer work with CMS configuration. No new database tables or backend services are required. The majority of effort is frontend (layout additions to the category template) with a content setup task for CMS block/widget configuration.

Phase 2 introduces a dependency on a brand data model and sales velocity data, which may require backend work outside the scope of the CLP template itself.

Key lever

Phase 1 effort is largely content setup, not engineering. Category hero images, SEO copy, and featured product curation are content team deliverables that must be ready before Phase 1 can launch — they are not on the dev critical path but can block go-live if not prepared in parallel.
