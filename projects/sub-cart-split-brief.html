Differentiate Schedule & Save
from Add to Cart on PDP

A focused PDP enhancement to surface the value of scheduled delivery subscriptions — using visual differentiation, contextual pricing, and a promo banner — to lift Subscribe & Save adoption and grow recurring revenue.

Platform: Magento Surface: Product Detail Page Delivery: Single phase Version 1.0
Problem statement

Pet Heaven's Schedule & Save programme offers customers meaningful savings on recurring orders — yet the PDP presents both purchase options (one-time Add to Cart and scheduled subscription) with near-identical visual weight. Customers scanning the page at speed have no clear signal that subscribing saves them money, and no friction-free path to understand the difference.

The result is that many customers who would gladly commit to a monthly bag of dog food — the highest-value, highest-retention segment — default to a one-time purchase because the value proposition of S&S was never surfaced at the decision moment. Once a customer has checked out as a one-time buyer, the window to convert them to a subscriber closes.

For repeat-purchase categories like pet food (where the same 25kg bag of Montego is repurchased every 6–8 weeks), a well-communicated S&S offer is a conversion lever, a retention tool, and a logistics planning asset. The PDP is the last high-intent screen before checkout — and currently it does almost no work to sell the subscription.

Acceptance criteria

1.1 — S&S option visual tint

  • When the S&S option is selected, the radio container receives a distinct background colour (brand-aligned, not default grey)
  • When the one-time option is selected, the S&S container returns to its unselected state with no tint
  • The tint passes WCAG AA contrast against the label text inside the container
  • Tint renders correctly on mobile viewport (<768px) without overflow or clipping

1.2 — S&S pricing callout

  • The regular price is shown with a strikethrough decoration when S&S option is active
  • The S&S price is displayed in a visually distinct treatment (larger, coloured, or bold) adjacent to the struck-through RRP
  • A savings amount or percentage is shown inline (e.g. "Save R90" or "22% off")
  • Pricing values are driven by the Magento product/rule data — not hardcoded in template
  • If no S&S price differential exists for the product, the callout does not render

1.3 & 1.4 — S&S promo banner + copy

  • The promo banner renders in the PDP info column, below the product description
  • Banner contains: deal headline, original and S&S price, promo code chip, first-order qualifier text
  • Tapping/clicking the promo code chip copies the code string to the clipboard using the Clipboard API
  • A visual confirmation state (e.g. green "Copied!" label) appears for 2 seconds then resets
  • Banner only renders when a valid promo code is configured for the product — CMS-driven flag, not hardcoded
  • Banner is responsive and does not overflow on 320px viewport

1.5 — Distinct S&S CTA

  • When S&S option is selected, the primary CTA reads "Subscribe & Save" (not "Add to Cart")
  • When one-time purchase is selected, the CTA reads "Add to Cart"
  • The CTA label change is driven by the selected radio state — no page reload required
  • The correct cart action fires based on the selected option regardless of label state
Key decisions
Promo code delivery method — CMS vs hardcoded Open

Whether promotional codes shown on the PDP banner are managed via a Magento CMS block (flexible, content-team controlled) or baked into the template per product (faster to build, harder to maintain). The CMS block route requires a small block-lookup pattern in the template; the hardcoded route creates future tech debt.

Option A — CMS block per SKU (recommended)
Content team can update codes without dev involvement
Banner disappears automatically when block is unpublished
Supports multiple products with different codes
Requires CMS block naming convention to be established
Small additional template logic (~10 lines)
Option B — Product attribute
All data in one place (product record)
Easy to query and render via standard Magento attribute pattern
Content team must access product admin to update
Banner logic coupled to product record rather than campaign layer
S&S radio visual treatment — CSS only vs component rebuild Locked

The S&S tint and border state will be implemented via CSS class-toggle on the selected radio container. No rebuild of the option selector component is required. This minimises risk to the existing add-to-cart flow and keeps the change isolated to a stylesheet addition.

Clipboard copy — Clipboard API with fallback Locked

The promo code copy action uses navigator.clipboard.writeText(). This API requires HTTPS (satisfied by petheaven.co.za) and is supported in all modern browsers. No Flash or legacy execCommand fallback is required given the SA ecommerce browser profile. A silent failure on unsupported browsers (no confirmation state appears) is acceptable given negligible exposure.

Constraints & dependencies
Tech Clipboard API requires HTTPS. All production Pet Heaven pages are served over HTTPS — no blocker. Local development environments may require localhost exception or manual test via staging.
Business The first-order promotional code (e.g. SSMONTEGO) must be created and configured in Magento's shopping cart price rules before the banner ships. Banner without a live code creates customer expectation that cannot be fulfilled.
Business The qualifier "1st scheduled order only" must be legally reviewed if used as a promotional term on-page. Marketing/legal sign-off required before copy is finalised.
Tech S&S pricing data (the discounted recurring price) must be exposed as a product attribute or price rule in Magento. If S&S pricing is handled outside Magento (e.g. a third-party subscription platform), an API lookup or attribute sync will be needed to render the correct savings figure inline.
UX The PDP info column layout must be audited for spacing before the banner is inserted. On products with long variant selectors or extensive descriptions, the banner position may need to be adjusted to remain above the fold on mobile.

Development effort estimate

All features are PDP-template-scoped. No new data models, no new API endpoints, no backend service changes. Effort is concentrated in frontend (template + CSS) with a small backend task to expose S&S pricing as a renderable attribute.

Key lever

If S&S pricing is managed natively in Magento's price rule layer, backend effort drops to near-zero. If a third-party subscription engine (e.g. Recharge, Bold) manages pricing, a sync mechanism must be added — this adds 3–5 days backend effort.
